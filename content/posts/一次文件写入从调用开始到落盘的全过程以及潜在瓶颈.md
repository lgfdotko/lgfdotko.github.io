# Overview
整体调用路径如下：

```mipsasm
write(fd, buf, count)
  → 系统调用入界（特权级切换、KPTI 开销）
  → VFS：fd 查表 → struct file → f_op->write_iter 分发到 ext4
  → ext4：inode 上锁、写检查 → 走 buffered 分支
  → page cache 写入循环：
      write_begin（锁页、预留块）→ 拷贝用户数据 → write_end（标脏、解锁）
  ── syscall 返回，数据仅在内存 ──
  → flusher 唤醒（定时 / 脏页阈值 / 显式 sync）
  → ext4_writepages：扫描连续脏页 → 分配物理块 → 构造 bio
  → 块层：bio 合并 → I/O 调度 → blk-mq 分发到驱动
  → NVMe 驱动：命令入 SQ → doorbell
  → 设备 DMA + 写入 NAND → CQ 完成 → 中断/polling → 清除 PG_writeback
```

# 库函数与系统调用
### 机制
应用程序需要调用库函数打开并修改文件，标准库的`open`，`write`，`fopen`，`fwrite`等接口内部包含了系统调用接口（`open`/`write` 是 POSIX 薄包装，每次调用直接进内核；`fopen`/`fwrite` 是在它们之上再包一层用户态缓冲），这些系统调用接口会把应用程序对文件的修改请求的参数传递给内核，由内核负责下一步执行。

库函数对应用程序的写文件请求做了一些优化，例如库函数会自己维护一段缓冲区，来尝试合并应用程序对文件的写入请求（`fwrite`），只有当满足某些条件（例如缓冲区满或者应用程序显式要求flush）才会真正进行系统调用（`write`），让内核进行文件写入。

### 瓶颈
- `fwrite`要自己维护缓冲区，就意味着需要把应用程序要写入文件的数据拷贝一份，对于频繁的小写入而言，用拷贝代替系统调用或许是一个好的选择；但是如果应用一次性要写入的文件较大时，继续采用拷贝方式反而会带来较大开销，这种情况下直接进行系统调用比较合理
- 系统调用与返回会导致处理器特权级切换，系统调用需要把用户态进程的上下文（通用寄存器,PC等）保存到内核栈空间上，然后才会进入到内核执行；当系统调用返回时，又需要把栈上保存的数据恢复到上下文中。这部分额外的load/store操作，系统调用参数检查等会带来额外的处理器周期消耗
- 如果系统为了安全开启了内核页表隔离（KPTI），那么上述过程还会发生页表的切换，CPU会刷TLB，进一步增加访存开销

# VFS与文件系统
### 机制
`write` 系统调用进入内核后，内核首先要找到文件描述符（fd）对应的内核文件对象。每个进程维护一张文件描述符表，fd 就是这张表的数组索引，通过它可以直接拿到对应的 `struct file` 结构体。这个结构体中保存了当前的文件读写偏移量、文件操作函数指针表以及指向文件元数据（inode）的指针等信息。

VFS 本身是一个抽象层，它不了解任何具体文件系统的实现细节。它完成通用检查（权限验证、文件大小限制等）后，通过 `struct file` 中的函数指针将写请求派发给该文件所在的具体文件系统。例如 ext4 文件系统注册的写入函数是 `ext4_file_write_iter`。

具体文件系统拿到写请求后，主要完成地址映射与数据拷贝两件事。以 ext4 为例，由于默认开启了延迟分配（delayed allocation, delalloc），buffered write 路径下此时**并不真正分配物理块**：内核通过 inode 的 extent tree 查找该逻辑偏移**是否已有**物理映射（覆盖写的情况能查到现成的块号），对于尚无映射的新块，则只做一次**延迟预留**——即在内存中记录一笔 reservation 并扣减配额，并不触碰块分配位图、也不修改 extent tree。真正的物理块分配、位图更新与 extent tree 修改，统统推迟到回写阶段的 `ext4_writepages` → `ext4_map_blocks(..., CREATE)` 才发生。这样做的好处是：攒够一批脏页再一次性分配，文件系统能挑到更连续的物理块，显著减少碎片。随后内核通过 `copy_from_user` 将用户缓冲区的数据拷贝到对应的 page cache 页中，并将这些页标记为脏页。至此 `write` 系统调用即可返回——数据并未落盘，而是暂存在内存的 page cache 里（除非应用在打开文件时显式使用 `O_DIRECT` 等 flag 要求绕过 page cache 直达块设备；但绕过 cache 并不保证持久化，持久化仍需 `O_DSYNC` 或调用 `fsync`）。

### 瓶颈
- VFS 层本身的开销很小，主要是数组索引查找文件描述符和一次函数指针跳转，通常不会成为瓶颈
- 文件系统的地址映射需要查找或修改 extent tree，当文件碎片化严重时，这个树的深度和复杂度会增加查找开销
- 多个线程并发写同一文件时，inode 锁会将写操作串行化，成为并发瓶颈

# Page Cache 写回
### 机制
数据被写入page cache后，系统调用就返回了，如果没有要求同步的话不会产生真实的I/O落盘。Linux 用的是一个 **per-BDI（Backing Device Info）的工作队列**。每个块设备（如 `/dev/sda`）拥有一个 `bdi_writeback` 结构体，里面挂了一个 delayed_work —— `wb_workfn`。触发它运行的时机有4种：
- 周期性唤醒：后台线程定期唤醒来执行脏页写回
- 后台阈值：全局脏页比例超过一定数值时，任何产生新的脏页的进程都会顺手唤醒flusher让它赶紧写回
- 前台节流：当脏页比例继续上升时，此时产生脏页的进程会被阻塞陷入睡眠，直到flusher把脏页比例降下来
- 显式同步：用户程序通过系统调用`fsync`等要求数据直接写入，不经过flusher队列，数据真正落盘后调用才会返回（`sync`：把全系统所有文件已缓存的修改（包括文件和元数据）写回硬盘；`fsync`：只对当前文件进行写回；`fdatasync`：只写回数据和必要的元数据，跳过非必要元数据，例如修改时间）

确定了哪些脏页要写回后，文件系统会根据之前建立的文件逻辑块->块设备物理块的映射，来决定脏页具体要写入硬盘的什么位置。在这个过程中，文件系统会把连续脏页合并到一个`bio`中（一个`bio`描述的是磁盘上一段连续的 LBA 范围，即起始扇区+长度，而内存端通过 `bio_vec` 数组可以引用多个不连续的页面），然后把这些封装好的`bio`提交给块设备层（`blk-mq`）。如果目标物理磁盘块之前没有分配过，则需要在此时分配

### 瓶颈
- 如果page cache中的脏页比例过高，flusher会阻塞当前产生脏页的进程，直到flusher的写回让脏页比例降低到一定阈值下，这个阻塞行为会降低产生写请求进程的吞吐，并且由于进程被阻塞可能会导致上下文切换（通用寄存器、pc、页表基地址切换，并且新进程的工作数据可能还不在Cache中，访存性能受到影响）
- 使用`fsync`显式同步时，需要等到数据真正落盘才会返回，因此设备也会成为瓶颈（例如需要把数据从块设备内部Cache中刷到持久化介质，内部需要读-改-写，或者块设备需要响应较多I/O请求时）
- 在写回阶段，对于延迟分配遗留下来的、尚无物理映射的脏页，文件系统需要真正执行块分配：查找并更新块分配位图，同时为保证崩溃一致性需要写入日志（journal），这些都是额外的开销。

# 块设备层调度与块设备驱动
### 机制
`submit_bio()` 把 bio 送入块设备层。现代内核用 **blk-mq**（多队列块层）架构（代码在 `block/blk-mq.c`），核心思想是每个 CPU 有自己的**软件队列（software queue）**，映射到设备的一个或多个**硬件队列（hardware queue）**。NVMe SSD 可能有 64 条硬件队列，每个 CPU 核心直接对应一条，完全无锁并行。

但 bio 不是直接入队的——内核会先尝试**合并**。如果队列里已经有一个 request，它的扇区范围和新 bio 刚好相邻（前面紧挨或后面紧挨），就把这个 bio 并入那个已有 request，而不是新建一个。这叫 **bio merge**，可以大幅减少发给设备的请求数。合并不了的 bio，才会被包装成一个新的 `struct request`，插入队列。注意区分这两个概念：`struct bio` 是一段连续 I/O 的描述，`struct request` 是发给设备的一个请求，一个 request 里可以挂一条 bio 链表（多个被合并的 bio）。

在软件队列和硬件队列之间，可以插入一个 **I/O 调度器**来决定以什么顺序把请求发给设备。现代内核（5.0+）可选 **mq-deadline**、**bfq** 或 **none**（代码在 `block/mq-deadline.c` 和 `block/bfq-iosched.c`）。调度器做两件事：

**排序**：把请求按扇区号排序。对机械硬盘来说，磁头按顺序扫过去比来回跳效率高得多。对 SSD 来说排序的意义不大，所以 NVMe SSD 通常用 **none**（不排序，直接发）。

**防饿死**：纯按扇区排序可能导致某些请求永远排不到（不断有更优位置的新请求插入）。所以 mq-deadline 维护两个队列——一个按扇区排序，一个按提交时间排序（FIFO）。请求等待超过截止时间（读默认 500ms，写默认 5s）就强制调度，防止饿死。

调度器选中一个 request 后，`blk_mq_dispatch_rq_list()` 调用设备驱动注册的 `.queue_rq()` 回调。对 NVMe 来说就是 `nvme_queue_rq()`（`drivers/nvme/host/pci.c`）。这个函数把 request 翻译成一条 **NVMe 命令**，写入硬件的 **Submission Queue**（一块设备和 CPU 共享的内存环形缓冲区），然后写 doorbell 寄存器通知设备："有新命令了。"写 doorbell 是一次 MMIO 操作，不可缓存，开销较大，因此可以攒多条命令后只写一次 doorbell，摊薄开销。

设备控制器（比如 NVMe 控制器）做完 DMA 传输后，往 **Completion Queue** 中写入一条完成记录，然后触发**中断**通知 CPU。在高 IOPS 场景下，频繁中断会带来显著开销，内核可以切换为 **polling** 模式，由 CPU 主动轮询 Completion Queue，以消除中断开销。

中断处理函数（`nvme_irq()`）或 polling 路径扫描 Completion Queue，找到完成的 request，调用 `blk_mq_complete_request()`，最终回调到 `bio->bi_end_io()`——这是文件系统在构造 bio 时注册的完成回调。

对 ext4 来说，这个回调（`ext4_end_bio()`）会清除相关页的 `PG_writeback` 标志，唤醒可能在 `wait_on_page_writeback()` 上阻塞的进程。至此，一个脏页从 page cache 到磁盘的完整旅程结束。

### 瓶颈
- 设备通过中断方式通知主机 I/O 任务完成，中断处理会打断当前 CPU 的执行流，带来寄存器保存、流水线刷新和缓存污染等开销。尤其是在 CPU 处理压力较高的小 I/O 场景下，这些额外负担会迅速堆叠，断崖式降低 CPU 效率。因此高 IOPS 场景下可切换为 polling 模式（例如用户态驱动）以消除中断开销。
- 设备驱动需要通过 MMIO 写设备寄存器（doorbell）来通知设备有新的请求产生，这种 MMIO 写操作是不可缓存的，因此频繁的 MMIO 也会降低 CPU 处理效率。合理的做法是攒多条命令后只做一次 doorbell 写入（batching），摊薄 MMIO 开销。
- 在开启了 IOMMU 的系统中，设备发起的 DMA 请求需要经过 IOMMU 的地址翻译才能访问内存。如果 DMA 请求的地址较为分散，可能会使用多条 I/O 页表项，造成 IOTLB Miss 率提高。这个问题可以通过使用大页（hugepages）来缓解。
